反馈系统与关键字识别界面说明
================================

本说明文档对应你看到的那几张界面截图，主要包括几部分：
1. 反馈列表与反馈统计界面
2. 反馈配置面板（工作流反馈配置 / 节点反馈配置 / 永久反馈）
3. 关键字识别规则管理与“全部关键字统计”弹窗

目的是：当你以后再看到这些界面时，不需要重新读代码，就能知道“它们在干什么、底层大概怎么工作、什么时候该用它们”。


一、反馈列表与反馈统计界面
--------------------------

反馈系统的所有文件，都落在 `fankui_log/` 目录下。`feedbackManager.js` 会在工作流或事件执行完成后，根据你配置的提示词自动生成三类文件：

- 工作流级反馈：对整个工作流的总结，按“工作流名 + 事件名 + 时间戳”归档。
- 节点级反馈：对单个视图 / 单个节点的表现做点评，按“事件名 + 文件名 + 视图ID + 时间戳”归档。
- 其他辅助反馈：例如你手动生成的某些复盘文件，也会跟着归档在同一层级，便于统一浏览。

你截图里的“反馈列表”页面，大致由三块组成：

1. 顶部筛选条  
   - 可以按时间范围筛选（天/小时），只看最近一段时间的反馈。  
   - 这背后会触发对 `fankui_log` 目录的扫描，并在前端做简单的统计聚合。

2. 反馈统计区块  
   - 显示当前筛选条件下的“总计反馈数 / 工作流反馈数 / 节点反馈数”。  
   - 这部分对应 `feedbackManager.js` 里对文件名模式的解析逻辑：根据命名约定判断“这是哪个事件、哪个工作流、哪个视图”的反馈。

3. 反馈文件列表  
   - 每一行对应一个实实在在的反馈文件（`.md` 或 `.txt`），上面是人类可读的标题，下面是完整路径。  
   - 点击文件行，会打开“反馈文件全屏编辑器”；点击右侧的垃圾桶图标，则可以删除该条反馈文件（注意这是直接删文件，而不是移到回收站）。


二、反馈文件全屏编辑器
----------------------

当你在反馈列表中点击某一条反馈文件时，会弹出全屏编辑器，这部分全部由 `feedbackManager.js` 管理：

1. 顶部信息行  
   - 标题：把文件名中的时间戳格式化成人类可读的时间，例如“2026年03月01日 22:16 · 某某工作流 · 某个视图反馈”。  
   - 路径：展示完整文件路径，方便你在外部编辑器中直接定位。  
   - 索引：当前是“第几条反馈 / 总共多少条”，例如 `1 / 3`。

2. 中部内容区  
   - 打开时，会通过 `getFile(path)` 读取原始 Markdown 内容，再用 `markdownConverter` 和 `marked` 渲染成可编辑的富文本。  
   - 你在这里的修改，最终会被 `htmlToMarkdown` 转回 Markdown，再落回原文件。  
   - 这意味着：你可以把反馈看成“AI + 自己的手写批注”，而不是只读文件。

3. 底部/右上角操作  
   - 保存：点击“保存”按钮，会把当前编辑内容写回原路径，按钮会短暂变成“✅ 已保存”。  
   - 关闭：点击“关闭”或按下 `ESC`，退出全屏编辑器，回到反馈列表。

4. 键盘导航（只在反馈编辑器里生效）  
   - `W`：跳到上一条反馈文件。  
   - `S`：跳到下一条反馈文件。  
   - `ESC`：关闭编辑器。  
   这些按键事件通过在捕获阶段监听 `keydown` 实现，不会被主界面的 `keyboardHandler` 抢走，所以在查看大量反馈时，你可以只用键盘快速浏览和微调。


三、反馈配置面板（全局控制反馈行为）
----------------------------------

你截图里那一页，就是“反馈配置”面板（有“工作流反馈配置”“节点反馈配置”等标题），它决定了反馈系统在生成反馈时到底用哪些提示词、读多少历史反馈，以及哪些视图有“永久反馈”加成。

这一块的核心状态都集中在 `feedbackManager.js` 里的 `feedbackConfig` 对象中，包括：

- `workflowFeedbackPrompts`: `{ workflowName: promptName }`，每个工作流对应一个“工作流级反馈提示词”。  
- `nodeFeedbackPrompts`: `{ viewId: promptName }`，每个视图（节点）对应一个“节点级反馈提示词”。  
- `feedbackCount` / `workflowFeedbackCount`: 控制生成新反馈时，会从历史中读取多少条工作流/节点反馈作为上下文。  
- `permanentFeedbacks`: `{ viewId: [filePath1, filePath2, ...] }`，指定某些“永久有效”的反馈文件，无论时间范围如何，总是参与该视图的反馈拼接。

界面的大致结构可以这样理解：

1. 顶部统计/模式区  
   - 这里会展示“反馈条数上限”、“工作流反馈读取数量”等输入框。  
   - 这些值直接映射到 `feedbackConfig.feedbackCount`、`feedbackConfig.workflowFeedbackCount` 和 `feedbackConfig.maxLimit`，通过保存按钮写入浏览器的 `localStorage`（键名为 `feedbackConfig`）。

2. 工作流反馈配置列表  
   - 每一行对应一个工作流（例如“测试工作流”“知识分析学习”等），右侧是一个下拉框，显示“未配置”或某个提示词名称。  
   - 选择某个提示词后，会把 `{ workflowName: promptName }` 写入 `feedbackConfig.workflowFeedbackPrompts`，并立即持久化到 `localStorage`。  
   - 生成“工作流级反馈”时，系统会优先查这里的配置，找到对应的提示词内容，作为 AI 生成整体反馈的 system / prompt 部分。

3. 节点（视图）反馈配置列表  
   - 每一行对应一个视图 ID，例如 `notes`、`summary`、`review` 等，同样通过下拉框选择提示词。  
   - 这部分映射到 `feedbackConfig.nodeFeedbackPrompts`，控制“节点级反馈”时该视图使用哪条提示词模板。  
   - 当某个视图在工作流中作为节点执行完毕后，如果这里有对应配置，就会为它单独生成一份“节点反馈”，文件名中会带上视图 ID。

4. 永久反馈配置  
   - 这部分界面允许你为某个视图挑选若干历史反馈文件，标记为“永久反馈”。  
   - 对应的数据结构是 `feedbackConfig.permanentFeedbacks[viewId] = [filePath1, filePath2, ...]`。  
   - 在后续每次生成该视图的新反馈时，这些永久反馈会被当成一种“长期经验”拼接进提示词中，而不受时间筛选和数量上限的限制。

5. 全屏反馈配置面板  
   - 当你点击某个“全屏配置”按钮时，会打开一个专门的全屏反馈配置界面（对应 `initFeedbackConfigFullscreenPanel` / `renderFeedbackConfigFullscreen`）。  
   - 这个面板内部依然是“工作流反馈配置列表 + 节点反馈配置列表”，只是把空间放大了一些，更适合在规则很多时集中编辑。  
   - 所有修改同样实时写回 `feedbackConfig` 并同步到 `localStorage`，你不需要担心两边配置不同步。

一个比较自然的使用方式是：

- 先在 `prompts/` 里写好“工作流级反馈提醒”“节点级反馈提醒”等提示词，然后在反馈配置面板里把它们挂到对应的工作流和视图上；  
- 按经验再决定“每次反馈前，要不要看 3 条历史反馈、还是只看 1 条”，以及为哪些关键视图绑定几份“永久反馈”；  
- 之后你就可以专心跑事件 / 工作流，反馈系统会按这份配置自动生成带“历史脉络”的评语。


四、关键字识别规则管理界面
--------------------------

关键字识别的目标，是从大量反馈文本中，把你关心的“模式”用统一的格式标出来，比如：  
`<<收入确认错误>>`、`<<内控风险·审批环节>>` 之类，让后续的统计、搜索、向量化更有结构。

关键字识别规则管理界面（你截图中右侧那一大块），对应的是 `keywordRecognitionManager` + `usage/keywordStats.js` 这一组逻辑，主要有几部分字段：

1. 左侧规则列表  
   - 左侧树形/列表展示当前已有的识别规则（例如“默认关键字识别”）。  
   - 搜索框可以按规则名或描述过滤，方便在规则较多时快速定位。

2. 规则详情区  
   典型字段含义如下：
   - 规则标题：这条规则在人类视角下的名字，比如“默认关键字识别”、“风险点识别”等。  
   - 开头标识：关键字的起始符号，例如 `<<`。  
   - 结束标识：关键字的结束符号，例如 `>>`。  
   - 是否启用：勾选后，这条规则会参与关键字识别和统计；未勾选则完全忽略。  

3. 功能提示模板  
   - 这一块是“写给 AI 看的说明书”，告诉模型：  
     - 你希望它在生成反馈时如何使用这类关键字标记；  
     - 什么样的内容该被包在 `开头标识` 与 `结束标识` 之间。  
   - 这些提示会融入到对应的反馈提示词里，帮助 AI 主动产出结构化的标记，而不是事后再去人工加标签。

4. 功能函数管理（与代码生成模板联动）  
   - 这一块会和 `关键字识别管理功能函数生成模板.md` 协同使用：  
     - 你可以为某个规则配置一组“辅助函数”，在后续的脚本或工作流中直接调用；  
     - 例如：“根据关键字筛选反馈文件”、“按关键字维度做聚类”等。  
   - 这里更偏开发者向，适合你后续写自动分析脚本时参考。


四、“全部关键字统计”弹窗
------------------------

在关键字识别规则界面中，你可以点击“全部关键字统计”按钮（截图中箭头所指），会弹出一个居中的统计对话框，这部分主要由 `usage/keywordStats.js` 驱动：

1. 基本选项  
   - 按规则统计：勾选后，会同时使用所有“已启用”的识别规则，从反馈文件中提取关键字。  
   - 每库文件数：对应 `filesPerLib` 配置，默认是每 100 个反馈文件合成一个“关键字库”。  
     - 这里的“库”，可以理解为一组关键字的子集，后续可以分库做向量化或更精细的分析。

2. 统计目标  
   - 统计范围默认是当前项目下的所有 `fankui_log` 目录，你也可以在界面上限制到某个子目录。  
   - 模块会扫描所有符合命名规则的反馈文件，从中提取形如 `开头标识 + 关键字 + 结束标识` 的片段。

3. 统计结果  
   - 统计完成后，会在对话框中展示每个关键字及其出现次数，并把结果写入 `fankui_log/guanjianzi` 目录下的一些 JSON/文本文件，用作后续分析的数据源。  
   - 当你看到“暂无关键字统计”提示时，一般意味着当前筛选范围内还没有符合规则的关键字，或者你还没在反馈提示词里引导 AI 使用这些标记。

4. 配置持久化  
   - “每库文件数”等配置会被保存在浏览器的 `localStorage` 中（键名为 `keywordStatsConfig`），下次打开统计弹窗时会沿用你上次的设置。  
   - 这让你可以根据实际反馈量，慢慢调优“每库多少文件”这个颗粒度，而不必每次手动输入。


五、建议使用流程小结
--------------------

结合反馈系统和关键字识别，这几个界面的一个自然用法是：

1. 在工作流/节点的反馈提示词里，明确告诉 AI：用某种括号格式标记你关心的内容（例如风险点、收获、疑问等）。  
2. 在关键字识别规则管理界面里，为这些括号格式配置清晰的“开头/结束标识”，并勾选启用。  
3. 跑一段时间工作流后，到反馈列表里按时间或事件浏览几份反馈，用全屏编辑器微调 AI 的标注习惯。  
4. 点击“全部关键字统计”，查看这些关键字在最近一段时间内的分布，并在 `fankui_log/guanjianzi` 中积累一个可被脚本进一步消费的结构化语料库。  

这样一来，反馈系统不只是“看一眼 AI 怎么评价你”，而是逐渐长成一套可以被搜索、被统计、被自动分析的经验数据库。
